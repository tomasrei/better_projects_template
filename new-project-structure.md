# R + Typst Project Structure

Replaces Quarto for research projects. R handles all computation; Typst compiles
the document almost instantly because it only reads pre-generated plots and tables.

---

# Package opinions

This package is intentionally opinionated. The opinions are listed here so they
can be debated and revised — but until changed, they are enforced by the code.

## tinylog_output() requires tinylog_script()

It is not possible to use `tinylog_output()` without first calling
`tinylog_script()`. If `.tinylog_current_script` is not set, the function
errors immediately.

The reason is intentional: the package **forces you to describe what a script
does before it lets you register what it produces**. Skipping `tinylog_script()`
means skipping `data_source` and `description` — and an output file with no
documented origin or purpose is exactly the kind of silent technical debt this
package is designed to prevent. The mild friction of writing two lines of
metadata is the point.

## The registry is always a reflection of the last run

`_tinylog_proj.yaml` is **never edited by hand**. It is entirely owned by the
package and rewritten on every script run. This is a deliberate constraint:

- `tinylog_script()` resets the script's `outputs` to `none` and clears its
  `data_dictionary` entries at the top of every run. By the time the script
  finishes, the registry reflects exactly what that run produced — not an
  accumulation of past runs.
- If you remove a `tinylog_output()` or `tinylog_dict()` call from a script,
  the corresponding entry disappears from the registry the next time that
  script runs. No manual cleanup needed.
- The file should be listed in `.gitignore` for the same reason generated
  plots and tables are: it is a build artifact, recreatable by running the
  scripts.

The only information that persists across runs intentionally is `first_run` —
recorded once and never overwritten, so you always know when a script was
first executed on this machine.

Future versions will allow **custom fields** via a user-supplied function, so
you can inject metadata that the package cannot detect automatically (e.g. a
sample size, a model AIC, a data version string). Until then, do not edit the
file — edits will be overwritten on the next run.

# Things to implement (including improvements)

- Custom fields via a user-supplied function (e.g. sample size, model AIC, data version string) — see the "registry is always a reflection" opinion above.
- `tinylog_output()` is currently tested with `ggsave`, `save_tt`, and `write.csv`. Need to audit other common save functions (e.g. `gt::gtsave()`, `flextable::save_as_docx()`, `openxlsx::saveWorkbook()`). **The design goal is that `tinylog_output()` should be pluggable into any save function in the R universe.**

Reference: data dictionary concepts — https://data-dict.tidyverse.org (talk: https://youtu.be/jVqArueBFqA?si=6uI_RUpZF9ws83WG)

## Folder layout

```
project/
├── data/
│   ├── raw/            ← original data, never modified
│   ├── clean/          ← RDS files written by *_clean.R scripts
│   └── misc/           ← crosswalks, lookups, small reference files (read only)
├── scripts/
│   ├── functions.R     ← project-specific custom functions
│   ├── occupation_clean.R
│   └── occupation_analyse_wages.R
├── typst/
│   ├── main.typ
│   ├── plots/          ← R writes plots here (PNG/SVG)
│   ├── tables/         ← R writes tables here (CSV or Typst snippets)
│   └── data.json       ← R writes computed inline stats here (jsonlite)
└── _tinylog_proj.yaml  ← auto-updated when any script runs
```

No numbered prefixes on folders or scripts. Never use `00_`, `01_` etc.

### .gitkeep

Git does not track empty folders — only files. To make an empty folder show up
in the repository (so the structure is preserved when someone clones or uses
the template), each empty folder contains a file called `.gitkeep`. It has no
content and does nothing; the name is just a convention. Once R writes real
files into a folder, the `.gitkeep` is irrelevant and can be deleted or left.

### .gitignore

`.gitignore` tells git which files and folders to never commit. This matters
for two reasons: keeping sensitive data off GitHub, and keeping auto-generated
files out (they can always be recreated by running the scripts).

**What is ignored and why:**

| Path | Reason |
|---|---|
| `data/raw/` | original data is often sensitive or too large for git |
| `data/clean/` | generated by `*_clean.R`; recreatable |
| `typst/plots/` | generated by R; recreatable |
| `typst/tables/` | generated by R; recreatable |
| `typst/data.json` | generated by inline stats helper; recreatable |
| `_tinylog_proj.yaml` | generated by `tinylog_script()`; recreatable |

**Example:** you start a new project and run `occupation_clean.R`. It writes
`data/clean/occupation.rds` and updates `_tinylog_proj.yaml`. Neither file will
appear in `git status` — git ignores them. Only your scripts and `typst/main.typ`
get committed. A collaborator who clones your repo just runs the scripts to
regenerate everything else.

**What is NOT ignored:**

| Path | Reason |
|---|---|
| `scripts/` | code is always committed |
| `typst/main.typ` | the document is always committed |
| `data/misc/` | crosswalks and reference files are inputs, not outputs |

## Script naming

Always use `{dataset}_{role}.R`:

```
occupation_clean.R
occupation_analyse_wages.R
occupation_analyse_industry.R
```

- One clean script per dataset. One or more analyse scripts per dataset, each
  with a descriptive suffix if there are multiple.
- `_clean.R` always saves to `data/clean/` and does nothing else downstream.
- `_analyse.R` always loads from `data/clean/` — never re-cleans.

Run order: `_clean.R` → `_analyse.R`. Once clean data is on disk, only rerun
the clean script if you discover a problem with the cleaning.

## Intermediate data format

Default: **RDS**. Preserves R types (factors, dates, labelled vectors) with
no `col_types` fiddling on load.

```r
# In *_clean.R
saveRDS(dat_clean, here("data/clean/occupation.rds"))

# In *_analyse.R
dat <- readRDS(here("data/clean/occupation.rds"))
```

Use CSV only when the clean data needs to be readable outside R.

## functions.R

All project functions live here. Every script sources it at the top:

```r
source(here("scripts/functions.R"))
```

### Function naming

All personal functions use the `f_` prefix to distinguish them from package
functions at a glance. `f_write_stats()` is immediately recognisable as yours;
`write_stats()` could be anything.

Within that, use a **category prefix** that describes what the function does:

| Prefix   | Purpose                          |
|----------|----------------------------------|
| `plot_`  | ggplot2 figures                  |
| `tab_`   | tables (gt, flextable, etc.)     |
| `calc_`  | derived statistics or transforms |
| `clean_` | data cleaning helpers            |
| `read_`  | custom data loaders              |

Examples: `f_plot_age_dist()`, `f_tab_descriptives()`, `f_calc_attrition_rate()`

## Script registry

`_tinylog_proj.yaml` tracks all scripts in the project. It is **auto-updated**
whenever a script runs — no manual editing required.

A core design goal is that the registry is **both human and AI readable**: a
researcher can open `_tinylog_proj.yaml` and immediately understand what every
script does and what it produced, and an LLM can consume the same file to
answer questions about the project ("what outputs did mtcars_clean.R produce?",
"which scripts read from data/clean/?") without needing to read the source code.

### How it works

The `tinylog` package provides three functions:

- `tinylog_script()` — call at the top of every script; records metadata, timestamps, and resets outputs and data dictionary for this run.
- `tinylog_output(file)` — wraps any file path at save time; registers the path and returns it unchanged so it drops inline into any save call.
- `tinylog_dict(df)` — call after loading or cleaning a data frame; records column names and sample values. Auto-names the entry from the df variable name.

```r
# occupation_clean.R
library(tinylog)

tinylog_script(
  data_source = "SCB register 2010–2023",
  description = "Cleans and recodes SSYK occupation codes"
)

dat <- readRDS(here("data/raw/occupation.rds")) |>
  tinylog_dict()

saveRDS(dat, file = tinylog_output(here("data/clean/occupation.rds")))

dat |> tinylog_dict()
```

```r
# occupation_analyse_wages.R
library(tinylog)

tinylog_script(
  data_source = "data/clean/occupation.rds",
  description = "Wage regressions by SSYK group"
)

dat <- readRDS(here("data/clean/occupation.rds")) |>
  tinylog_dict()

ggsave(filename = tinylog_output(here("typst/plots/wage_dist.png")), ...)
```

The resulting `_tinylog_proj.yaml` looks like:

```yaml
$version: 0.1.0
$learn_more: https://github.com/tomasrei/tinylog
scripts:
  occupation_clean.R:
    data_source: SCB register 2010–2023
    description: Cleans and recodes SSYK occupation codes
    first_run: '2026-06-01 09:00'
    latest_run: '2026-06-27 10:11'
    script_runtime: 0.3 min
    n_files: 1
    outputs:
    - data/clean/occupation.rds
  occupation_analyse_wages.R:
    data_source: data/clean/occupation.rds
    description: Wage regressions by SSYK group
    first_run: '2026-06-01 09:01'
    latest_run: '2026-06-27 10:14'
    script_runtime: 1.2 min
    n_files: 1
    outputs:
    - typst/plots/wage_dist.png
data_dictionary:
  occupation_clean.R:
    dat:
      columns:
        ssyk: ['1111', '1112', '1113', '1114', '1121']
        year: [2010, 2010, 2010, 2011, 2011]
```

Each run overwrites the previous entry — outputs and data dictionary entries reflect exactly what the last run produced.

## Misc / reference files

Store crosswalks, lookups, and small reference datasets in `data/misc/`.
R reads these with `readxl::read_excel()`. R never writes to these files.
Each Excel file can contain multiple sheets.

Keep these files separate from `_registry.xlsx` — mixing them in a single
Excel file risks accidental overwrites.

## Inline data in Typst (JSON)

For computed values that appear inside paragraphs — counts, means, percentages —
write them to `typst/data.json` with `jsonlite` and read them in Typst with
`json()`. This keeps numbers in sync without copy-pasting.

### R side

Namespace stats by dataset to avoid key collisions when multiple analysis
scripts write to the same file.

```r
library(jsonlite)
library(here)

# Compute your stats
n_accidents <- dat |> filter(year == 2007, city == "Liverpool") |> nrow()
mean_severity <- dat |> filter(year == 2007) |> summarise(m = mean(severity)) |> pull(m)

# Build a named list — use auto_unbox = TRUE so scalars are not wrapped in arrays
stats <- list(
  accidents = list(
    n_liverpool_2007  = n_accidents,
    mean_severity_2007 = round(mean_severity, 1)
  )
)

# Read existing data.json (if any), merge new stats, write back
json_path <- here("typst/data.json")
existing  <- if (file.exists(json_path)) read_json(json_path) else list()
merged    <- modifyList(existing, stats)
write_json(merged, json_path, auto_unbox = TRUE, pretty = TRUE)
```

`auto_unbox = TRUE` is required — without it, single values become arrays
(`[1432]` instead of `1432`) and Typst cannot use them as numbers.

### Typst side

```typst
#let data = json("data.json")

In 2007, the number of car-accidents in Liverpool was
#data.accidents.n_liverpool_2007.
The mean severity score was #data.accidents.mean_severity_2007.
```

Access is dot-notation by key name. String values work directly; integer and
float values render as their numeric string representation in text context.
If you need formatting (e.g. thousands separator) it is easier to do it in R
before writing: `format(n_accidents, big.mark = " ")`.

## Typst integration

R writes output directly into the `typst/` folder so Typst picks up changes
on the next compile (or live-reload if using `typst watch`).

```r
# Saving a plot
ggsave(here("typst/plots/age_distribution.png"), width = 6, height = 4)

# Saving a table as CSV (then read with typst-tablex or similar)
write.csv(tab, here("typst/tables/descriptives.csv"), row.names = FALSE)
```

In `main.typ`, define the paths at the top for easy adjustment:

```typst
#let plots  = "plots/"
#let tables = "tables/"

#image(plots + "age_distribution.png")
```

## Template repo vs R package

Two options for making project setup fast and keeping shared functions in sync.

**GitHub template repo**
- Pro: one click / `gh repo create --template` gives you the full folder structure with skeleton files
- Pro: trivial to set up — just a repo with the right folders and starter files
- Pro: no dependency to manage in your projects
- Con: utility functions still get copied into every project's `functions.R` — template drift over time as you improve them
- Con: improvements to the template don't reach existing projects

**R package**
- Pro: `tinylog_script()`, `tinylog_output()`, `tinylog_dict()`, and other helpers live in one place — update the package and all projects get it on next `remotes::install_github()`
- Pro: can add a `create_project("my-project")` function that builds the full folder structure programmatically
- Pro: functions are documented, not floating in a `functions.R` you have to remember to update
- Con: every project now has an external dependency — someone else running your code needs the package installed
- Con: more upfront work to build and maintain

**Recommendation: use both together**
The two are not mutually exclusive. The folder structure belongs in a GitHub
template (that's just files). The utility functions belong in a small personal
package — even a private one installed via `remotes::install_github("username/rproj")`.
Keep a local `functions.R` for project-specific functions; it sources the package
rather than containing the shared boilerplate.

Template repo → instant skeleton. Package → shared functions without copy-paste.

## Checklist for a new project

- [ ] Create folder structure: `data/raw/`, `data/clean/`, `data/misc/`,
      `scripts/`, `typst/plots/`, `typst/tables/`
- [ ] Install `tinylog`: `remotes::install_github("tomasrei/tinylog")`
- [ ] Create `typst/main.typ` with path variables at top
- [ ] Add `.gitignore` entries for `data/raw/` and `data/clean/` if data is sensitive
- [ ] Add `_tinylog_proj.yaml` and `typst/data.json` to `.gitignore` if auto-generated files should not be tracked

# Package name candidates

The package is a **project script registry**: scripts self-register at the top,
outputs self-register at save-time, and the result is a `_registry.yaml` audit
trail of what each script does, when it last ran, and what it produced.

## Names that describe the functionality

| Name | Rationale |
|---|---|
| **`tinylog`** | **Top contender. Fits the tinyverse family alongside `tinyplot` and `tinytable`, signalling zero-dependency lightweight design. "log" is accurate — the package keeps a log of script runs and outputs. Verify: `available::available("tinylog")`.** |

## Design goal: zero dependencies

`tinyplot` and `tinytable` both import only base R packages, which is central
to the tinyverse identity. `tinylog` should aim for the same.

- The JSON/Typst inline stats helper will **not** be part of the package — removes the `jsonlite` dependency entirely.
- `here` can be replaced: use `rprojroot::find_root_file()` directly, or detect the project root via `.here` / `.Rproj` / `DESCRIPTION` in pure base R.
- `yaml` is the remaining question — see the dependency audit in the CRAN roadmap.
